# 1. Introduction

This AP specifies how the Domain's **Capabilities** — the abstract descriptions of what the Domain requires from external systems — are expressed.

Capabilities are **Domain-owned** and reside in the `Capabilities/` peer folder of the Core Unit, alongside `Domain/`, `SharedKernel/`, and `Application/` (AP-001 §6.5, AP-002 §5.1.3, AP-004 §8). This AP does **not** define an "Infrastructure" layer or an "Outgoing Interface"; those terms are retired by AP-002. Capabilities are abstractions in domain terms; their concrete realizations are Adapter Implementations (AP-007).

# 3. Terminology

The logical term **Capability** retains the meaning defined in AP-001 [3]: a domain-owned abstraction describing something the Domain requires, expressed in domain terms and independent of any technology.

* **Capability Abstraction** — The interface (Port) that expresses a Capability in domain terms. It declares the operations the Domain requires without prescribing any technology.
* **Capability Contract** — The inputs and outputs of a Capability Abstraction, expressed with domain types (Entities, Value Objects, and Shared Kernel types), that define the meaning of each operation independently of how it is realized.

---

# 4. Relationship to Ports & Adapters

## 4.1 Capabilities as Ports

In the Ports and Adapters architecture [6], **Capabilities are Ports**.

A Capability serves a dual purpose:
- It is the Domain's abstraction of what it requires
- It is the Port through which external implementations connect

There is no separate "Port" layer or interface. The Capability interface itself IS the Port.

## 4.2 Pattern Structure

```
┌─────────────────────────────────────┐
│          Domain                     │
│                                     │
│  ┌───────────────────────────────┐ │
│  │ IPaymentRepository            │ │  ← Capability (Port)
│  │ (Domain-owned interface)      │ │     Expresses what Domain needs
│  └───────────────────────────────┘ │
└─────────────────────────────────────┘
              ↑ implements
┌─────────────────────────────────────┐
│ SqlitePaymentRepository             │  ← Adapter
│ (Adapter Implementation, SQLite)    │     Concrete realization (AP-007)
└─────────────────────────────────────┘
```

## 4.3 No Additional Abstraction Needed

A common mistake is to introduce:
- A "Port" interface AND a "Capability" interface
- A "Service" interface wrapping the Capability
- Additional abstraction layers between Domain and Adapter

These are unnecessary. The Capability itself provides:
- **Abstraction** - Domain defines what it needs
- **Port** - Adapters connect through this interface
- **Inversion of control** - Domain owns the interface, adapters depend on it

## 4.4 Clarity of Terms

For this architecture:
- **Capability** = Port = Domain-owned abstraction
- **Adapter** = Concrete implementation of a Capability
- **No intermediate layer** between Capability and Adapter

# 5. Scope

In scope:

* Persistence, time, identity, messaging, external systems, expressed as domain-term abstractions.
* How capability contracts are shaped (inputs/outputs in domain terms).
* Placement within the Core Unit's `Capabilities/` folder.

Out of scope:

* Concrete technology (SQL, Redis, HTTP, brokers) — see AP-007.
* Incoming Implementations — see AP-003.

---

# 6. Expressing Capabilities

A Capability Abstraction SHALL declare only the operations the Domain requires, named in the ubiquitous language of the service.

* A Capability MUST be expressed in domain terms only.
* A Capability MUST NOT reference transport, persistence, or messaging technologies (AP-001 §6.3).
* A Capability MUST NOT depend on any Implementation.

## 6.1 Shaping the Contract

* A Capability Contract SHALL express its inputs and outputs using domain types — Entities, Value Objects, and Shared Kernel types (AP-004 §7) — never transport- or persistence-specific shapes.
* Operation names SHALL describe *what* the Domain requires (for example, `Save`, `FindById`, `Now`, `Publish`), not *how* a technology provides it (for example, not `ExecuteSql` or `SendHttp`).
* A Capability SHOULD be narrow: it SHOULD declare only the operations an actual Domain need requires, rather than mirroring the full surface of a technology.
* A Capability MAY use Shared Kernel types directly in its contract where that preserves the ubiquitous language (AP-001 §6.5, AP-002 §5.1.3).
* Technology-specific failures SHALL be surfaced as domain-meaningful outcomes or errors by the Adapter (AP-007); the Capability Contract MUST NOT expose technology-specific error types.

---

# 7. Ownership and Placement

* Capabilities are owned by the Domain and expressed in domain terms (AP-001 §6.5).
* Capabilities reside in the Core Unit's `Capabilities/` folder, a peer folder alongside `Domain/`, `SharedKernel/`, and `Application/` (AP-002 §5.1.3, AP-004 §8).
* Capabilities are **not** a separate Unit or layer, and are **not** placed inside the `Domain/` folder.
* The `Capabilities/` folder MAY reference `SharedKernel/`, but MUST NOT reference `Domain/`, `Application/`, or any external Unit (AP-002 §7).

---

# 8. Compliance

A service complies with this AP if:

* Every Capability is expressed as a domain-owned abstraction (Port) in domain terms, with no separate or intermediate abstraction layer (§4, §6).
* Capability Contracts use only domain types — Entities, Value Objects, and Shared Kernel types — for their inputs and outputs (§6.1).
* No Capability references transport, persistence, or messaging technologies, or depends on any Implementation (§6).
* Capabilities reside in the Core Unit's `Capabilities/` peer folder, not inside `Domain/` and not in a separate Unit (§7).
* The `Capabilities/` folder references only `SharedKernel/` and nothing outside the Core (§7).
* Concrete realizations of Capabilities are provided exclusively by Adapter Implementations (AP-007), not within the Capability definitions (§1, §4).
