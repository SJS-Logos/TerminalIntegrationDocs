# 1. Introduction

*(Stub — outline only.)*

This RFC specifies how the Domain's **Capabilities** — the abstract descriptions of what the Domain requires from external systems — are expressed.

Capabilities are **Domain-owned** and reside inside the Domain Unit (AP-001 §6.3, AP-002 §5.1). This RFC does **not** define an "Infrastructure" layer or an "Outgoing Interface"; those terms are retired by AP-002. Capabilities are abstractions in domain terms; their concrete realizations are Adapter Implementations (AP-007).

# 4. Scope

*(Outline.)*

In scope:

* Persistence, time, identity, messaging, external systems, expressed as domain-term abstractions.
* How capability contracts are shaped (inputs/outputs in domain terms).
* Placement within the Domain Unit's `Abstractions` Module.

Out of scope:

* Concrete technology (SQL, Redis, HTTP, brokers) — see AP-007.
* Incoming Implementations — see AP-003.

---

# 5. Expressing Capabilities

*(Outline — to be specified.)*

* A Capability MUST be expressed in domain terms only.
* A Capability MUST NOT reference transport, persistence, or messaging technologies (AP-001 §6.3).
* A Capability MUST NOT depend on any Implementation.

---

# 6. Ownership and Placement

*(Outline — to be specified.)*

* Capabilities are owned by the Domain and live in the Domain Unit (AP-002 §5.1).
* Capabilities are **not** a separate Unit or layer.
