# 1. Introduction

This AP expands the dependency rules of AP-002 (§7) into a complete, enforceable set of allowed and forbidden dependencies across all Units defined by this series.

It restates the AP-001 invariant that dependencies move **toward business meaning** (§8) and applies it to incoming Implementations (AP-003), the Domain (AP-004), Capabilities (AP-005), and Adapter Implementations (AP-007).

## 1.1 Relationship to Other APs

This AP is the **single source of truth** for dependency direction across the series. It consolidates and does not contradict the rules stated in AP-002 §7 (intra-Core references) and AP-003 §9 (Host Unit constraints); where those documents state a rule, this AP restates it in matrix form. If a discrepancy is found, this AP governs for dependency direction, and the other AP SHOULD be corrected to match.

# 3. Dependency Direction

The fundamental rule is directional: dependencies SHALL always move toward business meaning (AP-001 §8).

* Business meaning resides in the Domain. The Application, Domain, Shared Kernel, and Capabilities together form the **Core**.
* The Domain depends on nothing outside itself except the Shared Kernel and its own Capabilities (AP-001 §6.4, AP-004 §9).
* The Shared Kernel is the innermost element: it depends on nothing (AP-002 §5.1.2).
* Implementations (Host Units, Composition Units, and Adapter Implementations) depend on the Core; the Core MUST NOT depend on any Implementation (AP-001 §7, AP-002 §7).

The permitted directions are summarized in the matrix of §4.

---

# 4. Allowed and Forbidden Dependencies

The following matrix defines the permitted direction of references between every Unit category in the series. A cell reads **From (row) → To (column)**. `—` marks the diagonal (a unit does not reference itself as a dependency).

| From \ To | SharedKernel | Capabilities | Domain | Application | Host / Composition | Adapter Impl. |
| --- | --- | --- | --- | --- | --- | --- |
| **SharedKernel** | — | forbidden | forbidden | forbidden | forbidden | forbidden |
| **Capabilities** | allowed | — | forbidden | forbidden | forbidden | forbidden |
| **Domain** | allowed | allowed | — | forbidden | forbidden | forbidden |
| **Application** | allowed | allowed | allowed | — | forbidden | forbidden |
| **Host / Composition** | allowed | forbidden (use Application) | forbidden (use Application) | allowed | — | allowed |
| **Adapter Impl.** | allowed | allowed (implements) | forbidden (use Capability) | forbidden | forbidden | — |

Notes on selected cells:

* **SharedKernel row** — the Shared Kernel references nothing; it is the innermost element (AP-002 §5.1.2).
* **Host / Composition → Domain / Capabilities = forbidden (use Application)** — Transport Endpoints depend only on the Application layer of the Core (AP-003 §5.2.1). They reach business behaviour exclusively through the Application.
* **Host / Composition → Adapter Impl. = allowed** — composition wires concrete Adapters to Capabilities at the outermost edge (AP-003 §5.3, §5.7).
* **Adapter Impl. → Capabilities = allowed (implements)** — an Adapter realizes a Capability by implementing its Port (AP-005 §4, AP-007).
* **Adapter Impl. → Domain = forbidden (use Capability)** — an Adapter interacts with the Domain only through the Capability contract it implements, never by depending on Domain internals.
* **Core → Implementation** cells are all **forbidden**: the Core (SharedKernel, Capabilities, Domain, Application) never references a Host, Composition, or Adapter Unit.

---

# 5. Layering Rules

* The **Core** — Application, Domain, Shared Kernel, and Capabilities — depends on nothing outside itself (AP-001 §3, AP-002 §4.1).
* Within the Core, references move inward: Application → Domain → Capabilities → Shared Kernel, with the Shared Kernel depending on nothing (AP-002 §7).
* Implementations (Host Units, Composition Units, Adapter Implementations) MAY reference the Core; the Core MUST NOT reference any Implementation.
* No cyclic dependencies are permitted between Units at any layer.

---

# 6. Enforcement

The dependency directions of §4 SHALL be enforced mechanically by the build system, expressed per language:

* **C#** — project references are restricted to permitted directions. The Core project declares no reference to any Host, Composition, or Adapter project. Intra-Core folder directions (Application/, Domain/, Capabilities/, SharedKernel/) are enforced by namespace discipline and code review (AP-002 §7).
* **C++** — target dependencies and header visibility are restricted to permitted directions. The Core target declares no dependency on Host or Adapter targets; include-path discipline enforces intra-Core directions (AP-002 §7).
* Where available, an automated architecture test (for example, a fitness function asserting the matrix of §4) SHOULD be added so violations fail the build rather than relying solely on review.

---

# 7. Compliance

A service complies with this AP if:

* Every reference between Units conforms to the direction permitted by the matrix in §4.
* The Core (SharedKernel, Capabilities, Domain, Application) references no Host, Composition, or Adapter Unit (§3, §5).
* The Shared Kernel references nothing (§4).
* Host and Composition Units reach business behaviour only through the Application, never by depending on the Domain or Capabilities directly (§4).
* Adapter Implementations depend on the Domain only through the Capability contracts they implement (§4).
* No cyclic dependencies exist between Units (§5).
* The permitted directions are enforced by the build system (§6).
