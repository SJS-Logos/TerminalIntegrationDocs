# 1. Introduction

This AP specifies **Adapter Implementations** — the concrete realizations of Domain Capabilities (AP-005) using specific technologies.

An Adapter's sole responsibility is **translation between two scopes**:
- **Domain scope**: Domain terms, Shared Kernel Value Objects, domain concepts
- **Technology scope**: HTTP, SQL, message formats, external APIs

Given a Capability expressed in domain terms and a target technology, the translation is straightforward: map domain concepts to technology concepts. This AP defines the principles governing that translation.

Adapters reside outside the core, may reference the core, and MUST NOT be referenced by it (AP-002 §7, AP-006). They contain no business decisions (AP-001 §6.1) and are replaceable without changing the Domain or Application.

# 3. Terminology

The logical term **Capability** retains the meaning defined in AP-001 [3] and AP-005 [5].

**Adapter** — A concrete implementation of a Capability that translates between domain scope and technology scope.

**Domain Scope** — Concepts, types, and operations expressed in domain terms (Shared Kernel Value Objects, domain identifiers, domain operations).

**Technology Scope** — Concepts, types, and operations specific to a technology (SQL types, HTTP requests, message payloads, API clients).

**Translation** — The bidirectional mapping between domain scope and technology scope.

---

# 4. Adapter Responsibility

## 4.1 Single Responsibility

An Adapter's sole responsibility is **translation**:
- **Inbound**: Technology → Domain (e.g., SQL row → Shared Kernel Value Object)
- **Outbound**: Domain → Technology (e.g., Shared Kernel Value Object → SQL parameters)

## 4.2 No Business Logic

An Adapter MUST NOT:
- Make business decisions
- Contain business rules
- Validate business invariants (those belong in the Domain)
- Orchestrate operations (that belongs in Use Cases)

An Adapter MAY:
- Validate technology constraints (e.g., SQL field length, HTTP status codes)
- Handle technology-specific errors (e.g., connection timeouts, serialization failures)
- Implement retry logic, circuit breakers, or other resilience patterns

## 4.3 Obvious Translation

Given:
- A Capability interface in domain terms
- A target technology

The translation SHOULD be **1:1 and obvious**:
- Each domain operation maps to a technology operation
- Each domain parameter maps to a technology parameter
- The structure mirrors the Capability interface

**Example**:
```
Capability (Domain Scope):
  PaymentRecord Save(Money amount, string merchantId, PaymentStatus status)

Adapter (Technology Scope - SQL):
  PaymentRecord Save(...) {
      // Money → decimal amount, string currency
      // merchantId → VARCHAR parameter
      // PaymentStatus → INT parameter
      // Execute INSERT
      // Return PaymentRecord (with generated ID)
  }
```

The mapping is mechanical, not creative.

# 5. Language-Specific Considerations

## 5.1 Translation Idioms

The mechanical nature of translation means Adapters follow language idioms:

**C#**:
- Use language features (LINQ, async/await, records)
- Follow .NET conventions
- Technology concerns (connection strings, HTTP clients) initialized at composition root

**C++**:
- Use language features (RAII, move semantics, templates)
- Follow C++ Core Guidelines
- Technology concerns (database handles, HTTP clients) initialized in adapter constructor

**General**:
- The Capability interface is language-neutral in concept
- The Adapter follows the target language's conventions
- Both implement the same logical translation

## 5.2 Technology Initialization

Technology-specific initialization (connection strings, API keys, HTTP clients, database connections) happens:
- At **composition root** (C#, using DI container)
- In **adapter constructor** (C++, passed as parameters)

**Technology initialization MUST be done as close to the Adapter implementation as possible** (same repository/folder). This ensures:
- Technology concerns remain localized to the Adapter
- Dependencies and configuration are co-located with their usage
- Changes to technology choices affect only the Adapter Unit

The Domain and Application remain unaware of these details.

# 6. Placement and Structure

Adapters live outside the core in separate technology-specific Units (AP-002 §6.2).

## 6.1 Organization

Adapters MAY be organized:
- **By technology**: All persistence adapters in one Unit, all messaging adapters in another
- **By capability**: Each Capability's adapter in its own Unit
- **By context**: Related adapters grouped together

The choice depends on service complexity and team preference. The key constraint: **Adapters MUST NOT be referenced by the core.**

For .NET services, the per-technology organization required by §8 takes precedence: each default technology SHALL reside in its own per-technology Adapter Unit.

## 6.2 Examples

**By technology**:
```
Logos.Payment.Service.Infrastructure.Sqlite/
├── SqlitePaymentRepository.cs
├── SqliteTransactionRepository.cs
Logos.Payment.Service.Infrastructure.RabbitMq/
├── RabbitMqEventPublisher.cs
```

**By capability**:
```
Logos.Payment.Service.Infrastructure.PaymentRepository/
├── SqlitePaymentRepository.cs
Logos.Payment.Service.Infrastructure.EventPublisher/
├── RabbitMqEventPublisher.cs
```

# 7. Constraints

An Adapter:
- MUST implement a Capability interface (AP-005)
- MUST translate between domain scope and technology scope
- MUST NOT contain business logic or make business decisions
- MUST NOT be referenced by the core (Domain or Application)
- MAY use technology-specific libraries and patterns
- MAY implement resilience patterns (retry, circuit breaker, timeout)
- SHOULD have an obvious 1:1 translation from Capability to technology

---

# 8. Technology Selections (.NET)

The following technology selections apply to services built on the .NET platform. They are **default recommendations**, not mandates: a .NET service SHOULD use them, and MAY use an alternative where justified. They do not bind services built on other platforms (for example C++).

Each selection realizes a Domain Capability (AP-005) and SHALL reside in its own per-technology Adapter Unit (AP-002 §6.2). A selection MUST NOT leak its technology into the core (AP-001 §6.3).

| Capability | Default technology (.NET) | Adapter Unit | Notes |
| --- | --- | --- | --- |
| Persistence | Entity Framework | Persistence Adapter Unit | Realizes the Persistence Capability (AP-005). |
| Messaging | MassTransit v8 | Messaging Adapter Unit | Realizes the Messaging Capability (AP-005). Pinned to v8 (last Apache 2.0 release); see version note below. |

## 8.1 Persistence — Entity Framework

* Entity Framework is the default .NET persistence technology.
* It SHALL be confined to a Persistence Adapter Unit; the core MUST NOT reference it.

## 8.2 Messaging — MassTransit

* MassTransit is the default .NET messaging technology.
* Services are pinned to **MassTransit v8 — the last release published under the open-source Apache 2.0 license.** Later releases moved to a commercial license, so services remain on this version.
* It SHALL be confined to a Messaging Adapter Unit; the core MUST NOT reference it.
* Adopting a commercially licensed version is out of scope for this AP and is tracked separately.

These selections are outside the scope of the architectural principles. They are implementation conventions specific to .NET services within the organization.

---

# 9. Compliance

A service complies with this AP if:
- Each Adapter implements exactly one Capability
- Adapters contain only translation logic, no business decisions
- The core does not reference any Adapter
- Translation between domain and technology scope is direct and obvious
