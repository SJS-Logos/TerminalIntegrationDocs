# 1. Introduction

*(Stub — outline only.)*

This RFC expands the testing structure of AP-002 (§8) into a full testing strategy for the service.

It defines the categories of tests, what each verifies, and how test Units respect the dependency rules (AP-006). It applies the testability invariant of AP-001 (§11): business behaviour SHALL be verifiable without deployment.

# 3. Test Categories

*(Outline — to be specified.)*

* **Unit tests** — Domain and Application behaviour, no Implementation, no deployment. Unit tests SHALL cover edge cases, error conditions, and business rule boundaries.
* **Component tests** — a Unit or small cluster in isolation.
* **Adapter (integration) tests** — Adapter Implementations against real technology (AP-007).
* **Deployment (end-to-end) tests** — composition, delivery, and configuration. End-to-end tests SHALL primarily verify the good path (successful scenarios) through the entire system.

# 4. What Each Category Verifies

*(Outline — to be specified.)*

* Mapping of test categories to the Units of AP-002 through AP-007.

## 4.1 Test Focus by Category

**Unit Tests**:
- Verify business logic in isolation
- Cover edge cases, boundary conditions, and error paths
- Test domain rules, policies, and invariants
- Fast execution, no external dependencies
- Example: "Payment declined when amount exceeds limit", "Payment declined for negative amounts"

**Component Tests**:
- Verify a Unit or small cluster works correctly
- Test interactions between closely related components
- May use test doubles for external dependencies

**Adapter (Integration) Tests**:
- Verify Adapter implementations against real technology
- Confirm correct translation between domain and technology scope
- Test technology-specific error handling
- Example: SQL connection failures, serialization errors

**Deployment (End-to-End) Tests**:
- Verify the good path through the entire system
- Confirm successful composition and configuration
- Test primary user scenarios from entry point to outcome
- Example: "User authorizes payment successfully", "User retrieves payment details"
- SHALL NOT exhaustively test edge cases (those belong in Unit tests)

### Rationale

This distribution follows the test pyramid [5]:
- **Many unit tests** provide fast feedback on business logic and comprehensive edge case coverage
- **Fewer integration tests** verify technology boundaries
- **Few end-to-end tests** confirm the system works as a whole for primary scenarios

End-to-end tests are expensive (slow, brittle, require deployment). Testing all edge cases at this level would:
- Slow down the feedback cycle
- Increase maintenance burden
- Provide redundant coverage (edge cases already verified in unit tests)

Edge cases verified in unit tests remain valid when the system is composed — the business logic doesn't change based on deployment.

# 6. Test Pyramid

*(Outline — to be specified.)*

The test strategy follows the test pyramid [5]:

```
        /\
       /E2E\       Few — Good path scenarios
      /------\
     /Integration\  Some — Technology boundaries
    /------------\
   /  Unit Tests  \  Many — Edge cases, business rules
  /----------------\
```

**Unit Tests (Base)**:
- Broadest coverage
- Fastest execution
- Test all edge cases and boundary conditions
- Verify business logic in complete isolation

**Integration Tests (Middle)**:
- Moderate coverage
- Test Adapter implementations with real technology
- Verify translations are correct

**End-to-End Tests (Top)**:
- Narrowest coverage
- Slowest execution
- Verify primary good path scenarios
- Confirm system composition

The pyramid shape reflects both **volume** (many unit tests, few E2E tests) and **focus** (comprehensive edge case testing at the unit level, primary scenario verification at the E2E level).

This distribution maximizes fast feedback while minimizing slow, brittle tests.

---

# 7. Compliance

*(Outline — to be specified.)*

A test suite complies with this RFC if:
- Business logic (Domain and Application) is verified by unit tests without deployment
- Unit tests provide comprehensive coverage of edge cases, boundary conditions, and error paths
- End-to-end tests focus on good path scenarios through the entire system
- The test pyramid shape is maintained (many unit tests, few E2E tests)
- Test Units follow dependency rules (AP-006): reference inward only
