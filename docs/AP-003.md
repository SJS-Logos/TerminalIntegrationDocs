# 1. Introduction

This RFC specifies the structure of **incoming Implementations**: the code that receives an external stimulus and invokes the Application.

Incoming Implementations are realized as **Host Units**. Each Host Unit corresponds to a specific inbound transport (HTTP, message broker, CLI, gRPC, etc.) and contains:
- A transport-specific folder (for example `Controllers/` or `Consumers/`)
- A `Mappings/` folder that translates between transport payloads and Application Contract Models
- A `Configuration/` folder that wires the host to the Application
- The application entry point

Incoming Implementations are an **Implementation category** in the sense of AP-001. They are not a new architectural layer and MUST NOT contain business behaviour. Their sole responsibility is to translate between the outside world and the Application's Contract Models, and to compose the service for deployment.

This RFC does not introduce an "Interface" layer; it structures one side of the generic Implementation defined in AP-001 and specifies how Host Units organize transport-specific endpoints and mappings.

# 3. Terminology

This section defines terms specific to incoming implementations and Host Units. Logical terms (Application, Domain, Capability, Core, Implementation) retain the meanings defined in AP-001 and AP-002.

* **Host Unit** — A physical Unit (as defined in AP-002 §3) that realizes incoming implementations for a specific transport. A Host Unit contains transport-specific endpoint classes, mappings, configuration logic, and an application entry point.
* **Incoming Implementation** — The code that receives an external stimulus (a Message) and invokes the Application; an Implementation category in the sense of AP-001. Realized physically within Host Units.
* **Transport Endpoint** — The transport-specific code within a Host Unit that receives external stimuli and delegates to the Application. Depending on technology, this may be a controller, consumer, command handler, or similar type.
* **Message** — An incoming request, command, or event that a service handles. A Message carries the caller's input and typically corresponds to exactly one Use Case.
* **Use Case** — An Application-level operation, defined by a service that handles a single incoming Message (see §6).
* **Mapping** — The translation between a transport-specific payload or envelope and the Application's Contract Models.
* **Subscriber** — A Transport Endpoint that receives Messages from a messaging transport rather than a synchronous request.

# 4. Scope

In scope:

* Host Unit structure and organization for inbound interfaces
* Transport-specific endpoint classes (HTTP controllers, message consumers, CLI handlers, etc.)
* Mappings between transport payloads and Application Contract Models
* Configuration and dependency wiring needed to expose the Application
* Application entry points
* Message mapping / deserialization to Application Contract Models
* Requests and responses
* Contracts exposed to callers
* Subscribers to external messages

Out of scope:

* Business decisions (owned by the Domain, AP-001 §6.1)
* Infrastructure adapters and persistence concerns
* Outbound interfaces and outgoing/secondary adapters
* Capability abstractions (AP-005)
* Deployment topology and operational concerns

# 5. Host Unit Structure

Each inbound transport SHALL be realized as a dedicated Host Unit. A Host Unit is organized to contain all the code required to receive external stimuli via that transport and to compose a runnable service.

## 5.1 Host Unit Organization

A Host Unit SHALL be organized into the following folders:

```text
Logos.Payment.Service.HttpHost/
├── Controllers/                  # HTTP-specific endpoint classes
│                                 # → Depend only on Application (Core)
│                                 # → Invoke Use Cases with Messages
├── Mappings/                     # Request/response mapping classes
│                                 # → Translate between HTTP payloads and Contracts
├── Configuration/
│   └── ServiceConfiguration.cs   # Host wiring and middleware setup
│                                 # → Exposes Application through HTTP
└── Program.cs                    # Application entry point
```

For a message broker host:

```text
Logos.Payment.Service.MasstransitHost/
├── Consumers/                    # MassTransit-specific endpoint classes
│                                 # → Depend only on Application (Core)
│                                 # → Invoke Use Cases with Messages
├── Mappings/                     # Message/envelope mapping classes
│                                 # → Translate between broker payloads and Contracts
├── Configuration/
│   └── ServiceConfiguration.cs   # DI container registration, consumer registration
│                                 # → Configures MassTransit consumers
└── Program.cs                    # Application entry point
```

For a command-line interface:

```text
Logos.Payment.Service.CliHost/
├── Commands/                     # CLI-specific endpoint classes
│                                 # → Depend only on Application (Core)
│                                 # → Invoke Use Cases with Messages
├── Mappings/                     # Argument/result mapping classes
│                                 # → Translate between CLI inputs and Contracts
├── Configuration/
│   └── ServiceConfiguration.cs   # DI container registration, command registration
│                                 # → Configures CLI command routing
└── Program.cs                    # Application entry point
```

## 5.2 Folder Responsibilities

### 5.2.1 Transport-Specific Endpoints

Transport-specific endpoint classes SHALL reside in a transport-named folder within the Host Unit. Examples include:
- `Controllers/` for HTTP endpoints
- `Consumers/` for message broker subscribers
- `Commands/` for CLI handlers
- `Handlers/` for gRPC services

Transport Endpoints rely exclusively on the Application layer of the Core Unit. They:
- Receive transport-specific payloads or envelopes
- Delegate translation to `Mappings/`
- Invoke Use Cases through the Application's public API

Transport Endpoints MUST NOT bypass the Application layer or depend on technology-specific implementation details outside the Core's public Application API.

### 5.2.2 Mappings

The `Mappings/` folder contains classes or functions that translate between transport-specific payloads and the Application's Contract Models.

Mappings MAY be implemented with manual code or with technology-specific mapping libraries, but the mapping logic SHALL remain in the Host Unit.

Mappings MUST NOT contain business rules. They are purely translational.

## 5.3 Configuration

The `Configuration/` folder contains the logic that exposes the Application through the host transport. This includes:
- Dependency injection container registration for the host and Application
- Middleware configuration (for HTTP hosts)
- Transport-specific setup (consumer registration for MassTransit, command registration for CLI)
- Configuration loading and validation

Configuration ensures that Transport Endpoints receive the Application services they need while remaining isolated from business logic and transport-independent concerns.

Configuration logic MUST NOT contain business rules. It is purely mechanical wiring.

## 5.4 Dependency Flow

The Host Unit's dependency structure ensures proper isolation:

```text
Transport Endpoint ──> Mapping ──> Application Use Case
        ↓                 ↓                 ↓
    (in Host)         (in Host)         (in Core)
                                            ↓
                                       Domain
                                     (in Core)
```

* **Transport Endpoints** depend only on Mappings, Application Contract Models, and Use Cases
* **Mappings** depend on transport payload types and Application Contract Models
* **Application Use Cases** depend on Domain and any Core-defined abstractions
* **Configuration** wires the host to the Application via dependency injection or equivalent mechanisms

This structure ensures:
- Transport Endpoints are transport-specific but business-agnostic
- Mappings isolate transport-specific shapes from Application Contracts
- Application Use Cases are technology-agnostic
- The Core remains independently testable without any Host Unit

## 5.5 Entry Point

The Host Unit's entry point (e.g., `Program.cs` in C#, `main.cpp` in C++) SHALL:
- Load configuration
- Invoke the Configuration logic
- Start the transport (HTTP server, message broker listener, CLI parser)
- Handle graceful shutdown

## 5.6 Naming Conventions

Host Units SHALL follow the naming pattern `Organization.ServiceName.HostType` as defined in AP-002 §6.1. Examples:
- `Logos.Payment.Service.HttpHost`
- `Logos.Payment.Service.MasstransitHost`
- `Logos.Payment.Service.CliHost`
- `Logos.Loyalty.HttpHost`
- `Company.Ordering.GrpcHost`

The `HostType` suffix SHALL clearly indicate the inbound transport.

# 6. Messages and Use Cases

*(Outline — to be specified.)*

An incoming Message typically corresponds to exactly one **Use Case**. A Use Case is defined by a service that handles a single incoming Message.

## 6.1 Naming

* A Use Case service SHALL be named after the Message it handles, with the suffix `UseCase` appended. For example, a `Quote` Message is handled by a `QuoteUseCase` service.
* The naming SHALL follow the service's casing convention (AP-002 §6.1).

## 6.2 Concrete Implementation

* A Use Case service SHALL be implemented as a concrete type with concrete methods. It MUST NOT be derived from, or hidden behind, an abstraction (interface) introduced solely to enable substitution.
* Use Case services are **not** intended to be mocked. Tests SHALL exercise the concrete Use Case service; substitution belongs at the Capability boundary (AP-005), not at the Use Case.

## 6.3 Placement

* The Use Case service and its Messages (both incoming and response) SHALL reside in the `Contracts` Module of the Core Unit, within the `Application/Contracts/` folder.
* Use Cases are part of the Core, not part of any Host Unit. Host Units reference the Core and invoke Use Cases through their public API.

## 6.4 Read/Write Split

*(Outline — to be specified.)*

Contracts SHOULD normally be split into **read** and **write** contracts, following a CQRS-style separation:

* **Read (query) contracts** — Messages that retrieve information without changing business state. Their Use Case services return data and produce no business outcome.
* **Write (command) contracts** — Messages that request a change to business state. Their Use Case services delegate the decision to the Domain (AP-001 §6.1) and return the outcome.

This split is the usual, recommended organization. A service MAY keep contracts unsplit where the distinction adds no value, but SHOULD document the deviation.

## 6.5 Documentation

* Documentation comments SHALL be placed with the **Message definition** so that the Message remains the single documented contract. This applies to both incoming Messages and response Messages.
* Comments SHALL be formatted as required by the tool that generates the service's documentation — for example Swashbuckle (OpenAPI) on .NET, or Doxygen on C++.

# 7. Responsibilities

## 7.1 Transport Endpoint Responsibilities

Transport Endpoints are responsible for:
* Receiving external payloads or envelopes
* Invoking Mappings and Application Use Cases
* Enforcing transport-level concerns (HTTP status codes, message acknowledgment, protocol framing)
* Handling transport-specific errors (connection failures, timeouts, malformed requests)

Transport Endpoints MUST NOT:
* Contain business logic or make business decisions
* Bypass the Application layer

## 7.2 Mapping Responsibilities

Mappings are responsible for:
* Translating external payloads to Application Contract Models (Messages)
* Translating Application responses back to transport-specific formats

Mappings MUST NOT:
* Contain business logic
* Determine business outcomes

## 7.3 Configuration Responsibilities

Configuration logic is responsible for:
* Configuring the dependency injection container
* Loading and validating configuration
* Setting up transport-specific runtime concerns (HTTP pipeline, message broker connections)

Configuration MUST NOT:
* Contain business logic
* Contain conditional logic based on business rules

# 8. Dependency Coordination

## 8.1 Coordination Mechanisms

The mechanism by which incoming messages are routed to Application services is a **configuration concern** and is outside the scope of this RFC.

Incoming DTOs MAY be dispatched:
- Directly to Application services (direct instantiation)
- Through a mediator pattern
- Through a command/query bus
- Via dependency injection container

This choice MUST NOT affect:
- The design of the service contract
- The structure of Contract Models
- The Domain model
- The Application's use case definitions

### 8.1.1 Architectural Invariants

Regardless of coordination mechanism:
- Use Cases MUST remain concrete types (§6.2)
- The Application MUST depend only on the Domain
- The Domain MUST remain independent of all coordination concerns
- Contract Models MUST be independent of dispatch mechanism

### 8.1.2 Language-Specific Considerations

Different languages provide different coordination mechanisms:
- **C# / .NET**: Dependency injection containers are idiomatic
- **C++**: Mediator pattern or manual composition may be used
- **Other languages**: Follow language-specific conventions

The choice is deferred to implementation and SHOULD follow the idioms of the target platform.

# 9. Constraints

Host Units and their adapters are subject to the following constraints:

* Host Units MUST depend on the Core Unit; the Core MUST NOT depend on any Host Unit (AP-002 §7).
* Transport Endpoints MUST depend only on the Application layer of the Core (§5.2.1).
* Transport Endpoints MUST NOT contain business rules or state.
* Transport Endpoints MUST NOT determine business outcomes; all decisions are delegated to the Application/Domain.
* Mappings MUST be purely translational; they MUST NOT contain business logic (§5.2.2).
* Configuration logic MUST be purely mechanical wiring; it MUST NOT contain conditional business logic.
* Use Case services (defined in the Core) MUST be concrete and MUST NOT introduce a substitution abstraction; they are not intended to be mocked (§6.2).
* Coordination mechanisms (DI, mediator, command bus) MUST NOT affect Contract Models or Domain design (§8.1).
* A Host Unit MAY be deployed as an independent process or combined with other Host Units in a single process (AP-002 §1.1); the structure supports both.

---

# 10. Compliance

A service complies with this RFC if:

* Each inbound transport is realized as a dedicated Host Unit (§5.1).
* Host Units are organized into a transport-specific folder, `Mappings/`, `Configuration/`, and an entry point (§5.1).
* Transport Endpoints depend only on the Core's Application layer (§5.2.1, §9).
* Host Units reference the Core Unit; the Core does not reference any Host Unit (§9).
* Transport Endpoints contain no business logic (§7.1, §9).
* Mappings contain no business logic (§7.2, §9).
* Use Cases are defined in the Core's `Application/Contracts/` folder, not in Host Units (§6.3).
* Host Unit names follow the `Organization.ServiceName.HostType` pattern (§5.6).
