# 1. Introduction

*(Stub — outline only.)*

This RFC expands the Domain principles of AP-001 (§6.3) and the Domain Unit structure of AP-002 (§5.1) into implementation guidance.

It defines how business rules, entities, policies, services, and outcomes are organized within the Domain Unit, and where the Domain's Capability abstractions reside.

# 3. Terminology

*(To be specified. Reuse AP-001 terms: Entity, Value Object, Domain Service, Business Decision.)*

* **Outcome** — The business result produced by a Domain operation; the expression of a Business Decision (AP-001 §6.1).
* **Policy** — A named business rule or set of rules that governs whether and how a Business Decision is made.
* **Aggregate** — A cluster of Entities and Value Objects treated as a single consistency boundary for business operations.
* **Value Object** — An immutable, data-carrying Domain type identified by its values rather than by identity (AP-001). See §7.

# 7. Value Objects

*(Outline — to be specified.)*

A Value Object is an immutable, data-carrying Domain type identified by its values. Structurally, a Value Object differs little from a Message (AP-003): both are shapes of data. This RFC therefore defines Value Objects **as shared message/contract types** so that a single definition serves both the Domain and the Application's Contract Models.

## 7.1 Shared Definition

* A Value Object SHALL be defined once as a data-carrying type and reused, rather than duplicated between the Domain and the incoming/outgoing contracts (AP-003).
* The incoming and response Messages of the Application (AP-003 §5) SHOULD reference these shared Value Object types rather than redeclare equivalent shapes.
* The intent is to **reduce duplication** between Domain Value Objects and Contract Models.

## 7.2 Placement

* Shared Value Objects SHALL reside in the Domain Unit, in a `ValueObjects` Module that sits alongside the `Abstractions` (Capabilities) Module (AP-002 §5.1).
* Because they live in the Domain Unit, dependency direction is preserved: the Application and its contracts reference the Domain (inward); the Domain references nothing (AP-001 §6.3, AP-002 §7).

## 7.3 Purity

* A Value Object MUST remain a pure Domain type. It MUST NOT carry transport, persistence, or serialization concerns (framework attributes, wire formats, ORM mappings).
* Any serialization or transport binding SHALL be applied by the relevant Implementation (AP-003, AP-007), never within the Value Object definition.

## 7.4 Documentation

* Value Objects SHALL be documented at their definition, in the format required by the documentation tool (Swashbuckle/Doxygen), consistent with AP-003 §5.5.
