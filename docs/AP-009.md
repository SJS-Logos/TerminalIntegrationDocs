# AP-009: Read Models and Queries

**Author:** SJS
**Created:** 2026-07-10
**Supersedes:** None
**Depends on:** AP-001, AP-002, AP-004, AP-005, AP-006

# 1. Introduction

This AP specifies the **query side** of a service: how information is read from the service's state and returned to callers without invoking Domain business behaviour.

The write side of the architecture is fully specified by AP-001 through AP-007: business decisions are made in the Domain (AP-001 §6.1), expressed through Aggregates and Domain Services (AP-004), and persisted through Capabilities (AP-005) realized by Adapters (AP-007). That path deliberately routes every change through an Aggregate Root to protect invariants.

Reading does not make business decisions and does not mutate state. Routing every read through an Aggregate Root imposes cost without benefit and can distort the Aggregate design to serve presentation needs. This AP introduces **Read Models** and **Queries** as the read-side counterpart, so that reads are expressed directly and the write-side Aggregates remain focused on invariants.

This AP is language-neutral in its normative rules. C#-specific mechanics (for example, record types and projection idioms) are governed by the .NET implementation profile, not by this AP.

## 1.1 Nature of the Optimization

Read Models and Queries are a **read-path optimization**, not a new Domain concern. The write side could, in principle, answer every read by loading an Aggregate and mapping it; this AP declines to do so because it would force Aggregate shapes to serve presentation, load more state than a read needs, and route reads through invariant-protection machinery they do not require.

The optimization is applied **around** the Domain, not **within** it:

* Read Models and Queries reside in the **Application** (§6.1, §7), not the Domain.
* A Query **bypasses** Domain business behaviour entirely — it invokes no Domain Service, Aggregate operation, or write Capability (§7).
* Projection to Read Models occurs in a read-side **Adapter** (Technology scope, §8), without materializing an Aggregate.

The Domain therefore remains untouched and continues to own every business decision (AP-001 §6.1). The read side borrows only the Domain's **ubiquitous language** — Query Capabilities are named in domain terms (§8) and Read Models MAY reuse Shared Kernel Value Objects (§6) — without depending on Domain **types or behaviour**. This is the sense in which the read side is *near* the Domain without being *at* it.

# 3. Terminology

The logical terms **Application**, **Domain**, **Aggregate**, **Entity**, **Value Object**, and **Capability** retain the meanings defined in AP-001, AP-004, and AP-005.

* **Read Model** — An immutable, data-carrying type shaped for a specific read purpose. A Read Model expresses information for presentation or integration, not business state that governs decisions.
* **Query** — An Application operation that returns a Read Model (or a collection of Read Models) without invoking Domain business behaviour and without mutating state.
* **Projection** — The mapping that produces a Read Model directly from the underlying store, without materializing a Domain Aggregate.
* **Query Capability** — A Capability (AP-005) that expresses a read requirement in domain terms, realized by a read-side Adapter (AP-007).

---

# 4. Scope

In scope:

* Definition and placement of Read Models.
* Queries as read-only Application operations.
* Projection of Read Models directly from the underlying store.
* The dependency direction of the read side and its relationship to Capabilities and Adapters.

Out of scope:

* Business decisions and state mutation (write side, AP-001 §6.1, AP-004).
* Concrete query technologies (SQL, search indices) — see AP-007.
* Transport shaping of query responses — see AP-003.

---

# 5. Separation of Read and Write

* A Query MUST NOT make a business decision (AP-001 §6.1) and MUST NOT mutate state.
* A Query MUST NOT return a Domain Aggregate, Entity, or mutable Domain type. It SHALL return a Read Model.
* A write operation MUST continue to route through an Aggregate Root (AP-004 §6.3); the read side does not weaken that rule.
* The read side and write side MAY share the same underlying store, or MAY use separate stores. This is a deployment and technology decision (AP-001 §4.1), not an architectural requirement of this AP.

---

# 6. Read Models

* A Read Model SHALL be immutable and data-carrying, defined by its values.
* A Read Model SHALL be shaped for a specific read purpose rather than mirroring an Aggregate's internal structure.
* A Read Model MAY reuse Shared Kernel Value Objects (AP-004 §7) where that preserves the ubiquitous language, but MUST NOT depend on Domain Aggregates or Entities.
* A Read Model MUST remain a pure data type: it MUST NOT carry transport, persistence, or serialization concerns (AP-004 §7.3). Any such binding is applied by the relevant Implementation (AP-003, AP-007).

## 6.1 Placement

* Read Models SHALL reside in the Application folder of the Core Unit (AP-002 §5.1.4), alongside the Queries that return them.
* Read Models are Application concerns, not Domain concerns; they MUST NOT reside in the `Domain/` folder.

---

# 7. Queries

* A Query SHALL be an Application operation that accepts primitive parameters or a request Contract Model (AP-003 §5) and returns one or more Read Models.
* A Query SHALL obtain data through a Query Capability (§8) or a projection realized by a read-side Adapter; it MUST NOT embed technology-specific query logic in the Application.
* A Query MUST NOT invoke a Domain Service, an Aggregate operation, or any write Capability.

---

# 8. Query Capabilities and Projection

* A read requirement SHALL be expressed as a Query Capability in the Core Unit's `Capabilities/` folder (AP-005), stated in domain terms and independent of technology.
* A Query Capability's contract SHALL return Read Models, not Domain Aggregates.
* Projection from the underlying store to Read Models SHALL be performed by a read-side Adapter (AP-007). The Adapter maps store-specific shapes to Read Models directly, without materializing an Aggregate.
* A read-side Adapter MUST NOT expose a technology-specific query abstraction (for example a live query object) beyond its own boundary; the query mechanics remain inside the Adapter (AP-005 §6.1, AP-006).

---

# 9. Dependency Rules

The read side obeys the dependency matrix of AP-006 §4 without exception:

* Read Models and Queries live in the Application and depend only on the Domain, Shared Kernel, and Capabilities.
* Query Capabilities live in `Capabilities/` and reference only the Shared Kernel.
* Read-side Adapters live outside the Core, implement Query Capabilities, and MUST NOT be referenced by the Core.
* The Core MUST NOT depend on any read-side Adapter.

---

# 10. Compliance

A service complies with this AP if:

* Every read is expressed as a Query that returns a Read Model and makes no business decision or mutation (§5, §7).
* Read Models are immutable, purpose-shaped, and free of Domain Aggregate/Entity dependencies and of transport/persistence concerns (§6).
* Read Models and Queries reside in the Application folder of the Core Unit (§6.1).
* Read requirements are expressed as Query Capabilities and realized by read-side Adapters through projection, without materializing Aggregates (§8).
* The read side conforms to the dependency matrix of AP-006 §4 (§9).

---

# 11. Open Questions

The following decisions are deferred to review before this AP is promoted beyond Draft:

* Whether a dedicated read store (with synchronization from the write store) is permitted, recommended, or discouraged as a default (§5).
* Whether Query Capabilities are always required, or whether simple projections MAY be realized without a Capability abstraction for trivial reads (§8).
* The .NET default query technology and projection idiom, to be recorded as a technology selection (AP-007 §9) rather than in this language-neutral AP.
