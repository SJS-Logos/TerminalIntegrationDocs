# 1. Introduction

This RFC defines the physical structure of a service that realizes the logical architecture defined in AP-001 [3].

AP-001 defines architectural invariants — the logical areas (Application, Domain, Capabilities) and the direction in which dependencies must flow. It deliberately says nothing about projects, folders, or assemblies.

This RFC supplies that missing layer. It specifies how the logical areas of AP-001 are organized into concrete units of compilation, how those units may reference one another, and how they are named. Where AP-001 asks *what a service is*, this RFC answers *how the core of a service is arranged*.

This RFC covers only the terms AP-001 introduces: the Application, the Domain, the Domain's Capabilities, and the generic notion of an Implementation. It does not define how Implementations are structured internally; that is deferred to a later RFC (Section 9).

This RFC is language-neutral in its normative rules. Each rule is expressed in terms of abstract structural units. Examples use a single, language-neutral notation; casing and naming differences between languages are governed by the rule in Section 6.1 rather than by parallel examples.

This RFC also formally retires the legacy folder names that predate this series (`Api`, `Core`, `InfrastructureAbstractions`, `Infrastructure`, and `Worker`). Section 6 maps each legacy name to its successor.

# 3. Terminology

This section defines the physical terms used throughout this RFC. Logical terms (Application, Domain, Capability, Core, Implementation) retain the meanings defined in AP-001 [3].

**Solution**: The complete set of units that together constitute a single service, together with the metadata that describes how they are built.

**Unit**: The smallest independently compiled artifact. A Unit realizes exactly one logical area of AP-001. In C# a Unit is a project (assembly); in C++ a Unit is a library target.

**Module**: A named grouping of related types within a Unit, corresponding to a folder or namespace. Modules organize a Unit internally; they are not independently deployable.

---

# 4. Logical-to-Physical Mapping

Each logical area defined in AP-001 SHALL be realized by a physical Unit according to the following mapping.

| AP-001 logical area | Physical realization (Unit) | Notes |
| --- | --- | --- |
| Domain | Domain Unit | Contains business rules, Entities, Value Objects, Domain Services. |
| Capabilities | *(inside the Domain Unit)* | Capabilities are owned by the Domain and SHALL reside within the Domain Unit (see Section 5.1). They are **not** a separate Unit. |
| Application | Application Unit | Contains use cases and Contract Models (see AP-001). |
| Implementation | *(separate Unit, outside the core)* | Implementations are separate Units outside the core (see Section 5.3). Their internal structure is defined in a later RFC. |

## 4.1 A Note on the Word "Core"

AP-001 uses the term **Core** to mean the logical whole formed by the Application, the Domain, and the Domain's Capabilities.

This term MUST NOT be confused with the legacy folder named `Core`, which contained a shared kernel of cross-cutting utilities. That legacy folder is unrelated to the AP-001 Core and is out of scope for this RFC (see Section 6).

Because of this collision, no Unit defined by this RFC SHALL be named `Core`.

# 5. Unit Layout

A service SHALL be organized into the Units described in this section. Examples use a language-neutral notation. The prefix `Company.Service` is a placeholder for the actual organization and service name; each language applies its own casing convention per Section 6.1.

## 5.1 The Domain Unit

The Domain Unit realizes the Domain of AP-001 and owns its Capabilities.

The Domain Unit SHALL contain a Module dedicated to Capabilities. This Module holds the domain-owned Capability abstractions and nothing else. Capabilities SHALL NOT be placed in a separate Unit.

The Domain Unit MUST NOT reference any other Unit of the service.

```text
Company.Service.Domain
├── Entities/
├── ValueObjects/
├── Services/
└── Abstractions/   (Capabilities)
```

## 5.2 The Application Unit

The Application Unit realizes the Application of AP-001. It contains use cases and the Contract Models that describe their inputs, outputs, and messages.

The Application Unit MAY reference the Domain Unit. It MUST NOT reference any Implementation.

```text
Company.Service.Application
├── UseCases/
├── Contracts/
└── Behaviours/
	references → Company.Service.Domain
```

## 5.3 Implementation Units

Implementations are realized by one or more Units that reside outside the core.

An Implementation Unit MAY reference the core. The core MUST NOT reference any Implementation Unit.

This RFC does not define how Implementation Units are organized internally, how they realize Capabilities, or how they are composed. Those concerns are deferred to a later RFC (Section 9).

# 6. Naming Conventions and Legacy Retirement

## 6.1 Naming Conventions

Unit names SHALL be derived from the logical area they realize.

* A Unit name SHALL identify its logical role (`Domain`, `Application`).
* A Unit name MUST NOT reuse any retired legacy name (Section 6.2) as its canonical role name.
* A service SHALL adopt a single casing convention and apply it consistently to all Unit and Module names. This RFC does not mandate a particular casing.
* Where a language has an established convention, that convention SHOULD be followed: in C# the logical name is used directly (for example `Company.Service.Domain`, with the root namespace matching); in C++ the same name SHOULD be mapped deterministically to the project's chosen convention — commonly the lower-cased, delimiter-normalized `snake_case` form (for example `company_service_domain`) following standard-library precedent.

The illustrative names in Section 5 use the logical form. The mapping to a language-specific form SHALL be deterministic, so that a Unit's identity is recognizable across languages.

## 6.2 Legacy Folder Retirement

The following legacy folder names are retired by this RFC. Each SHALL be replaced by its successor.

| Legacy folder | Successor | Notes |
| --- | --- | --- |
| `Api` | Implementation Unit | An Implementation outside the core (Section 5.3); its structure is defined in a later RFC. |
| `Core` | *(out of scope)* | Was a shared kernel of cross-cutting utilities. Not the AP-001 Core (Section 4.1). Shared Kernel is out of scope for this RFC. |
| `InfrastructureAbstractions` | Domain `Abstractions` Module (Section 5.1) | Capabilities now live inside the Domain Unit. |
| `Infrastructure` | Implementation Unit | An Implementation outside the core (Section 5.3); its structure is defined in a later RFC. |
| `Worker` | Implementation Unit | An Implementation outside the core (Section 5.3); its structure is defined in a later RFC. |

# 7. Reference Rules

Unit references SHALL enforce the Dependency Rule of AP-001: dependencies move toward business meaning, and the core depends upon nothing outside itself.

The following reference direction is permitted within the core:

```text
Application ──▶ Domain
```

The following references are prohibited:

* Domain to any other Unit.
* Application to any Implementation Unit.
* Any core Unit to any Implementation Unit.

These rules SHALL be enforced by the build system, expressed as follows in each language.

**C#** — enforcement is by project reference. A project reference SHALL exist only in a permitted direction. The Domain project declares no project references to other service projects.

**C++** — enforcement is by target dependency and header visibility. A target SHALL link and expose the include directories of another target only in a permitted direction. The Domain target declares no dependencies on other service targets.

---

# 8. Testing Structure

The physical structure SHALL support the testability required by AP-001 [3].

* The Domain and Application Units SHALL be verifiable by tests that reference only those Units, without any Implementation and without deployment.
* Test Units SHALL follow the same reference rules as the Units they exercise (Section 7): a test Unit MAY reference inward, never the reverse.

This structure is consistent with the test pyramid [6].

---

# 9. Scope and Evolution

This RFC defines the structure of the core only: the Application Unit, the Domain Unit, and the Capabilities the Domain owns.

The internal structure of Implementations — including how Implementations realize Capabilities, how delivery mechanisms are organized, and how the core is composed at runtime — is outside the scope of this RFC and SHALL be defined in a later RFC.

---

# 10. Compliance

A service complies with this RFC if:

* Each core area of AP-001 is realized by the Units defined in Section 4.
* Capabilities reside within the Domain Unit and are not a separate Unit.
* No Unit is named `Core`, and no retired legacy name is used as a canonical role name.
* All Unit references satisfy the reference rules of Section 7.
* The Domain and Application Units are verifiable without any Implementation and without deployment.
