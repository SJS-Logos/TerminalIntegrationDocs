# 1. Introduction

This AP defines the physical structure of a service that realizes the logical architecture defined in AP-001 [3].

AP-001 defines architectural invariants — the logical areas (Application, Domain, Capabilities) and the direction in which dependencies must flow. It deliberately says nothing about projects, folders, or assemblies.

This AP supplies that missing layer. It specifies how the logical areas of AP-001 are organized into concrete units of compilation, how those units may reference one another, and how they are named. Where AP-001 asks *what a service is*, this AP answers *how the core of a service is arranged*.

This AP covers only the terms AP-001 introduces: the Application, the Domain, the Domain's Capabilities, and the generic notion of an Implementation. It does not define how Implementations are structured internally; that is deferred to a later AP (Section 9).

This AP is language-neutral in its normative rules. Each rule is expressed in terms of abstract structural units. Examples use a single, language-neutral notation; casing and naming differences between languages are governed by the rule in Section 6.1 rather than by parallel examples.

## 1.1 Structure Does Not Imply Deployment

The structure defined in this AP — separate Units for Domain, Application, and various Implementation concerns — describes **logical boundaries**, not deployment topology.

Multiple separate Units MAY be:
- Deployed as a single process
- Deployed as independent processes
- Combined in any configuration that serves operational needs

The separation into Units ensures modularity, testability, and technology independence (AP-001 §4.1). It does not mandate distributed deployment. Deployment decisions are optimizations made when operational requirements justify the overhead.

This AP also formally retires the legacy folder names that predate this series (`Api`, `Core`, `InfrastructureAbstractions`, `Infrastructure`, and `Worker`). Section 6 maps each legacy name to its successor.

# 3. Terminology

This section defines the physical terms used throughout this AP. Logical terms (Application, Domain, Capability, Core, Implementation) retain the meanings defined in AP-001 [3].

**Solution**: The complete set of units that together constitute a single service, together with the metadata that describes how they are built.

**Unit**: The smallest independently compiled artifact. A Unit realizes exactly one logical area of AP-001. In C# a Unit is a project (assembly); in C++ a Unit is a library target.

**Module**: A named grouping of related types within a Unit, corresponding to a folder or namespace. Modules organize a Unit internally; they are not independently deployable.

**Host Unit**: A Unit that realizes one or more incoming Implementations for a single inbound transport. Host Units are defined in detail in AP-003 [4].

**Adapter**: A component that translates between a specific technology and the Core. Adapters are Implementations in the sense of AP-001 and are defined in detail in AP-003 [4].

# 4. Logical-to-Physical Mapping

Each logical area defined in AP-001 SHALL be realized by a physical Unit according to the following mapping.

| AP-001 logical area | Physical realization (Unit) | Notes |
| --- | --- | --- |
| Domain | `Core/Domain/` folder | Business rules, Entities, and Domain Services within the Core Unit. |
| Capabilities | `Core/Capabilities/` folder | Domain-owned abstractions at same level as Domain and `SharedKernel/` (see Section 5.1). |
| Application | `Core/Application/` folder | Use cases and Contract Models within the Core Unit (see AP-001). |
| Implementation | Adapters within Host Units | Adapters reside within Host folders. Host Units are defined in AP-003 (Incoming Implementations). |

## 4.1 A Note on the Word "Core"

AP-001 uses the term **Core** to mean the logical whole formed by the Application, the Domain, and the Domain's Capabilities.

The physical structure defined by this AP realizes the Core as a **single Unit** named after the application or service. For example, a payment service would use `Logos.Payment.Service.Core`. The pattern is `Organization.ServiceName.Core`.

The Core Unit contains four peer folders:
- `Domain/` — Business logic (Entities and Domain Services)
- `SharedKernel/` — Shared immutable business datatypes used across the Core
- `Capabilities/` — Domain-owned abstractions (interfaces for external concerns)
- `Application/` — Use cases and contracts

This term MUST NOT be confused with the legacy folder named `Core`, which contained a shared kernel of cross-cutting utilities. That legacy folder is unrelated to the AP-001 Core and is out of scope for this AP (see Section 6).

# 5. Unit Layout

A service SHALL be organized into the Units described in this section. Examples use a language-neutral notation.

The prefix `Logos.Payment.Service` represents the organization and service name; the actual name is derived from the application or bounded context being implemented (e.g., `Logos.Payment.Service`, `Logos.Loyalty.Service`, `Company.ServiceName`). Each language applies its own casing convention per Section 6.1.

## 5.1 The Core Unit

The Core Unit realizes the Domain, Capabilities, and Application of AP-001 as a single compiled Unit. It is organized into four peer folders:

The Core Unit MUST NOT reference any external Unit.

```text
Core/
├── Domain/
│   ├── Entities/
│   └── Services/
├── SharedKernel/
│   ├── Money/
│   ├── Identifiers/
│   └── Dates/
├── Capabilities/
│   ├── Persistence/
│   ├── Messaging/
│   └── ExternalServices/
└── Application/
	├── UseCases/
	├── Contracts/
	└── Behaviours/
```

### 5.1.1 Domain Folder

The Domain folder contains business rules, Entities, and Domain Services.

The Domain folder MAY reference `SharedKernel/` and `Capabilities/`, but MUST NOT reference the `Application/` folder or any external Unit.

### 5.1.2 SharedKernel Folder

The `SharedKernel/` folder contains shared immutable business datatypes that form part of the service's ubiquitous language.

`SharedKernel/` exists as a top-level peer folder so that Domain, Capabilities, and Application can share the same business datatypes without introducing awkward bidirectional dependencies or duplicate transport-neutral models.

The SharedKernel folder MUST NOT reference Domain, Capabilities, Application, or any external Unit.

### 5.1.3 Capabilities Folder

The `Capabilities/` folder contains domain-owned interface definitions that external adapters must implement. It is at the same level as Domain to emphasize that:
- Capabilities are contracts defined by domain needs
- Capabilities are independent abstractions, not implementation details
- Both Domain and Application can reference these abstractions
- Capabilities MAY use `SharedKernel/` directly in their contracts when that preserves the ubiquitous language of the service

Capabilities contains only interface definitions and data contracts, no implementations.

The Capabilities folder MAY reference `SharedKernel/`, but MUST NOT reference `Domain/`, `Application/`, or any external Unit.

### 5.1.4 Application Folder

The Application folder contains use cases and Contract Models (inputs, outputs, and messages). Application MAY reference `Domain/`, `SharedKernel/`, and `Capabilities/`, but MUST NOT reference any external Unit.

## 5.2 Host Units (Incoming Implementations)

Host Units realize incoming implementations and are defined in detail in **AP-003: Incoming Implementations**.

Each inbound transport (HTTP, MassTransit consumers, gRPC, etc.) has its own dedicated Host Unit that contains:
- Transport-specific inbound adapter
- Composition logic
- Application entry point

For complete specification of Host Unit structure, naming, organization, and composition, see AP-003.

# 6. Naming Conventions and Legacy Retirement

## 6.1 Naming Conventions

Unit names SHALL be derived from the logical area they realize.

* The Core Unit SHALL be named to identify the service/bounded context following the pattern `Organization.ServiceName.Core` (e.g., `Logos.Payment.Service.Core`, `Logos.Loyalty.Service.Core`, `Company.Ordering.Core`).
* Host Units SHALL follow the pattern `Organization.ServiceName.HostType` (e.g., `Logos.Payment.Service.HttpHost`, `Logos.Payment.Service.MasstransitHost`). See AP-003 for details.
* A Unit name MUST NOT reuse any retired legacy name (Section 6.2) as its canonical role name.
* A service SHALL adopt a single casing convention and apply it consistently to all Unit and Module names. This AP does not mandate a particular casing.
* Where a language has an established convention, that convention SHOULD be followed: in C# the logical name is used directly (for example `Logos.Payment.Service.Core`, with the root namespace matching); in C++ the same name SHOULD be mapped deterministically to the project's chosen convention — commonly the lower-cased, delimiter-normalized `snake_case` form (for example `logos_payment_service_core`) following standard-library precedent.

The illustrative names in Section 5 use the logical form. The mapping to a language-specific form SHALL be deterministic, so that a Unit's identity is recognizable across languages.

## 6.2 Legacy Folder Retirement

The following legacy folder names are retired by this AP. Each SHALL be replaced by its successor.

| Legacy folder | Successor | Notes |
| --- | --- | --- |
| `Api` | Host Units with embedded adapters | Split into Host Units by transport (HttpHost, MasstransitHost). See AP-003. |
| `Core` | *(out of scope)* | Was a shared kernel of cross-cutting utilities. Not the AP-001 Core (Section 4.1). Shared Kernel is out of scope for this AP. |
| `InfrastructureAbstractions` | `Capabilities/` folder within Core Unit (Section 5.1.3) | Capabilities are now a peer folder alongside Domain, SharedKernel, and Application. |
| `Infrastructure` | Technology-specific Infrastructure folders or Units | Split into focused infrastructure implementations such as `Infrastructure.MsSql` rather than one monolithic `Infrastructure` folder. |
| `Worker` | Host Units with embedded adapters | Split into Host Units per transport, each containing its own adapters and composition. See AP-003. |

**Important:** The legacy `Worker`, `Api`, and `Infrastructure` folders were often monolithic, combining multiple technologies in one place. This AP replaces that centralization by splitting inbound mechanisms into Host Units (defined in AP-003) and infrastructure concerns into focused technology-specific infrastructure folders or Units.

# 7. Reference Rules

Unit references SHALL enforce the Dependency Rule of AP-001: dependencies move toward business meaning, and the Core depends upon nothing outside itself.

Within the Core Unit, the following internal reference directions are permitted:

```text
Application/ ──▶ Domain/
Application/ ──▶ SharedKernel/
Application/ ──▶ Capabilities/
Domain/ ──▶ SharedKernel/
Domain/ ──▶ Capabilities/         (Domain may reference its own capability interfaces)
Capabilities/ ──▶ SharedKernel/   (Capabilities may use shared business datatypes in their contracts)
```

The following references are prohibited:

* `SharedKernel/` to `Domain/`, `Capabilities/`, or `Application/`.
* `Capabilities/` to `Domain/` or `Application/`.
* `Domain/` to `Application/` (within Core).
* Core Unit to any Host Unit.

Host Units MAY reference the Core Unit. The Core Unit MUST NOT reference any Host Unit.

These rules SHALL be enforced by the build system, expressed as follows in each language.

**C#** — enforcement is by project reference and namespace organization. The Core is a single project. Folder-level dependencies within the Core (Domain/ vs SharedKernel/ vs Capabilities/ vs Application/) are enforced by namespace visibility and code review. The Core project declares no project references to Host projects.

**C++** — enforcement is by target dependency and header visibility. The Core is a single library target. Folder-level dependencies are enforced by include path discipline and header guards. The Core target declares no dependencies on Host targets.

# 8. Testing Structure

The physical structure SHALL support the testability required by AP-001 [3].

* The Core Unit (containing `Domain/`, `SharedKernel/`, `Capabilities/`, and `Application/` folders) SHALL be verifiable by tests that reference only the Core Unit, without any Implementation and without deployment.
* Test Units SHALL follow the same reference rules: tests MAY reference the Core Unit and exercise its public API, but the Core MUST NOT reference test Units.

As a suggested organization, test projects SHOULD follow the same Unit structure as production code and be separated by test category. For example:

```text
Logos.Payment.Service.Core/
Logos.Payment.Service.Core.UnitTests/

Logos.Payment.Service.Infrastructure.Sql/
Logos.Payment.Service.Infrastructure.Sql.IntegrationTests/

Logos.Payment.Service.HttpHost/
Logos.Payment.Service.HttpHost.IntegrationTests/

Logos.Payment.Service.EndToEndTests/
```

This suggestion is intended to preserve traceability between production Units and their tests while keeping test categories explicit. Unit tests typically align to Core Units, integration tests to technology-specific Units, and end-to-end tests to the fully composed service.

This structure is consistent with the test pyramid [7].

---

# 9. Scope and Evolution

This AP defines the structure of the Core: a single Core Unit organized into four peer folders (`Domain/`, `SharedKernel/`, `Capabilities/`, and `Application/`).

Host Units (incoming implementations) are defined in AP-003. The internal structure of transport-specific endpoints, mappings, how Hosts organize composition, and how configuration is managed are all specified in AP-003.

---

# 10. Compliance

A service complies with this AP if:

* The Core is realized as a single Unit organized into `Domain/`, `SharedKernel/`, `Capabilities/`, and `Application/` peer folders (Section 5.1).
* `SharedKernel/` is a peer folder at the same level as Domain, Capabilities, and Application, and contains shared immutable business datatypes (Section 5.1.2).
* Capabilities is a peer folder at the same level as Domain, SharedKernel, and Application, and MAY reference `SharedKernel/` while remaining free of implementations and external dependencies (Section 5.1.3).
* Host Units (incoming implementations) are structured according to AP-003.
* No retired legacy name is used as a canonical role name.
* Internal Core folder references satisfy the dependency direction defined in Section 7.
* The Core Unit is verifiable without any Host Unit and without deployment.
